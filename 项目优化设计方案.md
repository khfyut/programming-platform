# 个性化在线编程与答疑系统优化设计方案

## 1. 项目现状概述

### 1.1 基础技术栈

项目当前采用的基础技术栈较清晰，整体方向适合作为毕业设计和后续迭代基础：

- 前端：Vue 3、Vite、Element Plus、Pinia、Vue Router、Monaco Editor
- 后端：Spring Boot 3、MyBatis、MySQL、Spring Security、Docker Java API
- 基础设施：Docker、可选 Redis、WebSocket、SSE
- AI 能力：通过外部大模型接口接入问答与代码讲解能力

从工程角度看，这套技术栈已经能够支撑“在线编程 + 提交判题 + AI 辅导 + 学习支持 + 后台管理”的完整产品形态，不需要推翻重来。

### 1.2 已完成的核心能力

结合代码结构、接口文档和页面分布，当前项目已经具备以下核心模块：

- 用户注册登录、JWT 认证、管理员权限控制
- 题库浏览、题目详情、测试用例展示
- 在线代码运行、正式提交判题、提交记录查询
- AI 编程问答与代码解释
- 学习中心、学习路径、知识图谱、能力测评
- 错题本、社区、个人中心、后台管理

### 1.3 当前架构特征

项目采用典型的前后端分离架构：

- 前端以“页面驱动”为主，业务逻辑大量集中在单文件组件中
- 后端采用 Controller + Service + Mapper 的经典三层结构
- 在线运行与判题通过 `DockerUtil` 完成容器级代码隔离执行
- 提交判题支持 WebSocket 进度反馈
- AI 讲解能力通过 SSE 流式输出

整体上，项目已经不是一个简单 demo，而是一个功能较完整的综合平台。但从代码形态看，它仍处于“功能先跑通，工程化待加强”的阶段。

## 2. 现状分析

### 2.1 优势

#### 2.1.1 功能覆盖面完整

项目不是单一 OJ，而是融合了题库、判题、AI、学习路径、错题本、社区、后台等多个模块，适合作为“个性化在线编程与答疑系统”的毕业设计主题。

#### 2.1.2 技术路线统一

前后端技术选择一致，没有明显的多套技术混用问题。基础设施也围绕 Docker、MySQL、JWT 展开，便于维护与汇报。

#### 2.1.3 关键链路已打通

以下主业务链路已经具备实际可用性：

- 登录 -> 浏览题目 -> 编写代码 -> 运行 -> 提交 -> 查看结果
- AI 提问 -> 获取解释/建议
- 提交失败 -> 错题记录 -> 后续复盘
- 管理员维护题目、查看提交、管理用户

#### 2.1.4 有一定扩展基础

项目中已经存在 Redis 配置、全局异常处理、WebSocket、SSE、学习路径与知识点模型，这说明它具备继续往“更工程化、更智能化”方向演进的基础。

### 2.2 主要问题

#### 2.2.1 判题执行层稳定性仍然偏弱

当前判题链路依赖 `DockerUtil` 直接负责：

- 镜像选择
- 容器创建
- 代码写入
- 编译与运行
- 超时/内存控制
- 日志采集
- 容器删除
- 内存统计

这导致执行层职责过重，任何细节调整都容易产生回归。近期已经出现过：

- 转义字符导致 Java 代码编译失败
- 输入未真正写入标准输入
- 时间限制与内存限制单位不一致
- 日志抓取与容器等待存在竞态
- `output` 为空导致上层 `trim()` 报错

这说明当前判题模块“功能可用，但脆弱度较高”。

#### 2.2.2 Service 和 View 层存在明显的大文件问题

当前文件体量已经暴露出维护风险：

- `frontend/src/views/Admin.vue` 约 2179 行
- `frontend/src/views/ProblemDetail.vue` 约 1956 行
- `frontend/src/views/Home.vue` 约 1843 行
- `System/src/main/java/com/programming/service/impl/LearningServiceImpl.java` 约 546 行
- `System/src/main/java/com/programming/service/impl/AdminServiceImpl.java` 约 487 行
- `System/src/main/java/com/programming/controller/AdminController.java` 约 414 行

这类大文件通常意味着：

- 状态管理分散
- 业务逻辑耦合严重
- 局部改动容易引发连锁影响
- 测试难度高
- 新成员接手成本高

#### 2.2.3 学习中心模块存在“页面已做、业务未完全落地”的情况

`LearnServiceImpl` 中多个能力仍是占位实现或弱实现，例如：

- 能力测评题目获取
- 测评结果驱动学习路径
- 关卡解锁
- 路径进度
- 知识图谱
- 周报
- 弱知识点识别

因此，这一模块在展示层面比较完整，但算法和业务闭环仍偏浅。它适合在论文中表述为“已完成基础框架与部分功能落地”，而不应夸大为成熟的个性化推荐系统。

#### 2.2.4 查询层存在 N+1 和重复查询问题

从 `ProblemServiceImpl` 与 `LearnServiceImpl` 可以看到多个典型问题：

- 题库列表中逐题统计提交数与通过数
- 统计用户难度分布时逐条提交再查题目
- 推荐逻辑中多轮分散查询

在数据量较小时问题不明显，但当题目数、提交数增长后，会成为明显瓶颈。

#### 2.2.5 安全体系存在“Spring Security 形态在，但主认证依赖拦截器”的混合状态

当前 `SecurityConfig` 中采用 `anyRequest().permitAll()`，实际鉴权主要依赖：

- `JwtInterceptor`
- `AdminInterceptor`

这种方案能跑通，但存在几个问题：

- 认证入口分散
- 权限语义不统一
- 与 Spring Security 的整合价值没有真正发挥
- 后续扩展细粒度权限时成本较高

#### 2.2.6 缓存和异步能力储备有，但没有形成系统化设计

项目已经引入：

- Redis 配置
- WebSocket
- SSE
- 固定线程池

但目前仍缺少统一的：

- 缓存策略
- 异步任务边界
- 线程池配置管理
- 限流与熔断机制
- 观测指标

这意味着系统在“单机功能验证”层面可用，但距离“稳定服务化”还有差距。

#### 2.2.7 测试体系明显不足

项目中存在较多接口测试脚本和测试报告，但缺少：

- 后端单元测试
- Service 集成测试
- 判题链路回归测试
- 前端组件测试
- E2E 自动化测试

这也是为什么执行层一处改动后，容易带来多轮回归问题。

## 3. 优化目标

在不改变基础技术栈的前提下，项目优化应围绕以下五个目标展开：

### 3.1 提高核心链路稳定性

重点保障：

- 在线运行
- 正式提交判题
- 提交记录展示
- AI 答疑响应

### 3.2 提高代码可维护性

通过拆分大文件、统一 DTO/VO、提取公共逻辑、梳理分层边界，降低后续修改成本。

### 3.3 提高系统性能与可扩展性

围绕高频查询、判题链路和统计展示，优化数据库访问模式与缓存策略。

### 3.4 提高“学习支持”模块的真实落地度

不追求复杂算法堆砌，而是把现有学习路径、知识点、错题本、推荐逻辑做成“简单但真实可用”的闭环。

### 3.5 提高可测试性与可运维性

让项目不仅能演示，还能稳定迭代、方便答辩和后续扩展。

## 4. 总体优化思路

建议采用“保留基础栈 + 增强工程能力 + 收紧核心链路 + 补足学习闭环”的路线。

不建议：

- 重写前后端
- 更换 ORM
- 切换微服务
- 引入过重中间件体系

建议增加的技术与能力：

- 后端：Spring Cache、Resilience4j、Micrometer、OpenAPI/Knife4j、MapStruct、JUnit 5、Testcontainers
- 前端：TypeScript 渐进引入、VueUse、Pinia 按域拆 store、ECharts 统一图表层、Vitest、Playwright
- 运维：Docker Compose、Prometheus + Grafana（可选）、Nginx

这些增强不会改变基础技术栈，只是让现有架构更工程化。

## 5. 分模块优化方案

### 5.1 架构层优化

#### 5.1.1 后端分层再整理

建议将当前后端进一步规范为：

- Controller：只做参数接收、鉴权上下文获取、结果返回
- Application Service：负责编排流程
- Domain Service：封装核心业务规则
- Infrastructure：Mapper、Docker、AI、Redis 等外部依赖

在现有项目里，可以不强行引入完整 DDD，但建议至少做以下拆分：

- `SubmitServiceImpl` 拆为
  - `SubmitApplicationService`
  - `JudgeOrchestrator`
  - `SubmitRecordService`
- `DockerUtil` 拆为
  - `DockerSandboxClient`
  - `SandboxCommandBuilder`
  - `SandboxResultParser`
  - `ResourceLimitResolver`

收益：

- 判题链路定位更清晰
- 单点复杂类被拆散
- 更容易做测试和回归验证

#### 5.1.2 前端按“页面 + 领域组件 + 组合式逻辑”拆分

建议对超大页面做分层：

- 页面：只负责布局和路由参数
- 领域组件：如题目描述区、代码编辑区、提交记录区、提交详情弹窗
- composables：如 `useProblemDetail`、`useSubmissionList`、`useJudgeProgress`
- API 层：按领域划分

例如 `ProblemDetail.vue` 可拆成：

- `components/problem/ProblemDescriptionPanel.vue`
- `components/problem/CodeEditorPanel.vue`
- `components/problem/SubmissionHistoryPanel.vue`
- `components/problem/SubmissionDetailDialog.vue`
- `composables/useProblemDetail.js`
- `composables/useProblemSubmission.js`
- `composables/useJudgeSubmit.js`

### 5.2 判题与代码运行模块优化

这是项目最值得优化的核心模块。

#### 5.2.1 将判题拆成“编译阶段”和“运行阶段”

当前 Java 判题把：

- 容器准备
- 编译
- 运行

统合在一个时间窗口中，容易造成误判超时。

建议优化为：

1. 代码写入容器
2. 编译阶段单独执行并记录编译耗时
3. 运行阶段按测试用例执行并记录运行耗时
4. 评测结果中分别保存：
   - 编译耗时
   - 最大运行耗时
   - 峰值内存

这样既更符合 OJ 语义，也更便于问题定位。

#### 5.2.2 增加执行状态机

建议为执行结果设计明确状态：

- `PENDING`
- `COMPILING`
- `RUNNING`
- `COMPILE_ERROR`
- `RUNTIME_ERROR`
- `TIME_LIMIT_EXCEEDED`
- `MEMORY_LIMIT_EXCEEDED`
- `SYSTEM_ERROR`
- `SUCCESS`

这样 WebSocket 推送、数据库记录、前端展示会统一很多。

#### 5.2.3 引入沙箱适配层

保留 Docker 作为底层实现，但对外暴露统一接口：

- `SandboxExecutor`
- `SandboxExecutionRequest`
- `SandboxExecutionResult`

这样未来即使继续增加：

- 代码缓存
- 容器预热
- 多语言扩展
- 本地沙箱与 Docker 沙箱切换

也不会影响上层业务。

#### 5.2.4 资源控制增强

在现有内存限制与网络隔离基础上，建议进一步补充：

- CPU 配额限制
- 只读根文件系统
- 指定工作目录
- 非 root 用户运行
- 临时目录清理
- 单次提交总执行用例上限

新增技术建议：

- 继续使用 Docker Java API
- 可增加 `seccomp/apparmor` 配置
- 可增加镜像预校验机制

#### 5.2.5 容器池作为中期优化，而非当前必选项

项目当前不是容器池模式。对于毕设阶段，不建议优先上池化，因为复杂度较高且回归成本大。

更合适的顺序是：

1. 先把当前“按次创建、按次销毁”模式做稳定
2. 再考虑“镜像预热 + 编译缓存 + 容器池”

如果后续要做容器池，建议仅对“运行容器”池化，不对业务逻辑大幅重构。

### 5.3 数据层优化

#### 5.3.1 消除高频 N+1 查询

重点优化以下场景：

- 题库列表统计通过率
- 学习统计页难度分布
- 用户提交记录关联题目信息
- 推荐逻辑的题目聚合查询

建议改法：

- 用聚合 SQL 一次查出统计数据
- 将用户维度状态统一批量查出
- 对题库列表使用分页 + 批量附加统计

例如问题列表不应对每道题单独查提交统计，而应通过一条聚合查询返回：

- `problem_id`
- `total_submissions`
- `passed_submissions`

#### 5.3.2 补索引设计

建议重点检查并新增以下索引：

- `submit(user_id, problem_id, create_time)`
- `submit(problem_id, result)`
- `submit_test_case_result(submit_id)`
- `test_case(problem_id, sort_order)`
- `favorite(user_id, target_type, target_id)`
- `wrong_book_item(wrong_book_id, problem_id)`
- `knowledge_mastery(user_id, knowledge_point_id)`

#### 5.3.3 引入缓存策略

建议在以下数据上使用 Redis + Spring Cache：

- 题库语言列表
- 热门题目列表
- 题目详情基础信息
- 学习路径结构
- 知识点图谱静态结构

缓存策略建议：

- 热点读多写少数据：缓存 5 到 30 分钟
- 题目更新后精确清缓存
- 用户个人数据不做长缓存，避免一致性复杂化

### 5.4 学习中心与个性化能力优化

这一块不建议追求“复杂推荐算法”，而是建议做成“规则驱动的真实闭环”。

#### 5.4.1 统一学习画像模型

建议为用户建立简化学习画像：

- 最近 30 次提交正确率
- 各难度题通过率
- 各知识点练习次数
- 各知识点错误次数
- 最近学习活跃度
- 偏好语言

可将其汇总为一张用户画像表，定时更新或提交后增量更新。

#### 5.4.2 将推荐逻辑从“占位”改为“规则引擎式推荐”

不引入复杂 AI 推荐系统，直接使用规则组合：

- 优先推荐错题关联题
- 其次推荐弱知识点题
- 再推荐当前路径下一关卡题
- 最后补充同难度热门题

这样可解释性强，答辩时也更稳。

#### 5.4.3 学习路径闭环化

建议补齐：

- 关卡完成条件
- 解锁条件
- 当前路径进度计算
- 用户学习路径状态表
- 每关推荐资源关联

这样学习路径就不再只是“展示页面”，而是真正的学习流程。

### 5.5 AI 模块优化

#### 5.5.1 AI 能力边界更明确

建议将 AI 能力拆为三类：

- 题目思路提示
- 代码解释与错误分析
- 知识点讲解

避免一个接口包办所有场景，导致提示词和返回格式不可控。

#### 5.5.2 增加 AI 结果治理

可以新增：

- 提问频控
- 敏感词与越权内容过滤
- 返回内容长度限制
- 失败重试与降级
- AI 会话摘要与归档

推荐增加技术：

- Resilience4j：做外部 AI 调用的超时、重试、熔断
- Spring Retry：简单重试场景也可使用

### 5.6 前端体验优化

#### 5.6.1 统一请求与错误处理机制

当前前端已经有统一 `request.js`，这是好基础。建议继续增强：

- 统一 loading 状态
- 统一接口错误 toast 策略
- 业务错误与系统错误区分展示
- 请求取消与防重复提交

#### 5.6.2 状态管理按业务域拆分

目前 Pinia store 较少，很多状态留在页面内部。建议至少补齐：

- `authStore`
- `problemStore`
- `submissionStore`
- `learnStore`
- `uiStore`

这样跨页面共享状态会更清晰。

#### 5.6.3 渐进引入 TypeScript

不需要全量重构前端，但建议从以下层开始：

- API 返回类型
- 提交结果模型
- 题目详情模型
- 学习路径模型

可先把 `src/api` 和 `src/composables` 改为 TS，再逐步扩展。

### 5.7 安全与权限优化

#### 5.7.1 将权限校验逐步收口到 Spring Security

不需要立刻废弃拦截器，但建议中期目标是：

- JWT 解析进入 Security Context
- 普通接口通过 `authenticated()`
- 管理接口通过角色控制
- 方法级权限可用 `@PreAuthorize`

这样权限模型会更统一。

#### 5.7.2 补输入校验

当前已有全局异常处理，但校验体系还不充分。建议：

- VO 层增加 `jakarta.validation` 注解
- Controller 方法使用 `@Valid`
- 统一非法参数返回格式

重点接口：

- 登录注册
- 提交代码
- AI 提问
- 管理端新增/编辑题目

### 5.8 可观测性与运维优化

#### 5.8.1 日志结构化

建议统一日志字段：

- traceId
- userId
- problemId
- submitId
- executionId
- language
- result
- timeCost
- memoryCost

可先通过 MDC 实现，不需要引入过重链路追踪系统。

#### 5.8.2 增加指标采集

推荐增加：

- Spring Boot Actuator
- Micrometer

重点监控指标：

- 接口响应时间
- 判题成功率
- 编译错误率
- 执行超时率
- AI 调用失败率
- WebSocket 活跃连接数

可选增加：

- Prometheus
- Grafana

### 5.9 测试体系优化

#### 5.9.1 后端测试

建议优先补三类测试：

- `DockerUtil` 相关判题回归测试
- `SubmitServiceImpl` 提交流程测试
- `ProblemServiceImpl` / `LearnServiceImpl` 聚合查询测试

建议新增技术：

- JUnit 5
- Mockito
- Testcontainers

其中 Testcontainers 特别适合这个项目，因为它能直接在测试里拉起 MySQL 或 Docker 依赖环境，更贴近真实链路。

#### 5.9.2 前端测试

建议新增：

- Vitest：做工具函数、composables、核心组件逻辑测试
- Playwright：覆盖登录、题目提交、查看记录等主流程

## 6. 推荐新增技术栈

在不改变基础技术栈的前提下，推荐新增如下技术：

### 6.1 后端推荐新增

- Spring Cache：统一缓存抽象

这个项目的基础技术路线是成立的，功能覆盖也足够支撑“个性化在线编程与答疑系统”这一题目。当前最需要的不是更换技术栈，而是把已有能力做扎实：

- 复杂类拆开
- 查询做轻
- 学习能力做成闭环
