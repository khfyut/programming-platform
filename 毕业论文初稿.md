# 个性化在线编程与答疑系统的设计与实现

## 摘要

随着编程教育的普及，在线编程学习平台已经成为编程课程教学和自主学习中的重要工具。传统在线编程平台虽然能够提供题目练习和基础判题能力，但在个性化学习支持、即时答疑反馈、学习过程跟踪以及综合教学辅助等方面仍存在不足。针对上述问题，本文设计并实现了一个个性化在线编程与答疑系统，旨在为学习者提供集在线编程、自动判题、AI 辅导、学习路径管理、错题复习、学习社区和后台管理于一体的综合性学习平台。

本系统采用前后端分离架构实现，前端基于 Vue 3 和 Vite 构建，结合 Element Plus 与 Monaco Editor 实现交互界面与在线编码环境；后端基于 Spring Boot 搭建业务服务，使用 MyBatis 完成数据访问，采用 MySQL 存储系统业务数据，并通过 JWT 实现用户身份认证与权限控制。在代码执行与判题场景中，系统引入 Docker 容器技术对用户代码进行隔离运行，以提高系统安全性和运行稳定性；在智能辅导场景中，系统接入大语言模型接口，实现编程问答、多轮会话、代码优化建议和知识点讲解等功能。

围绕平台核心业务，本文完成了用户与权限管理模块、题库与题目管理模块、在线代码运行与自动判题模块、学习中心与学习路径模块、错题本与提交记录模块、AI 智能答疑模块、社区交流模块以及管理后台模块的设计与实现。数据库层面构建了覆盖用户、题目、测试用例、提交记录、学习路径、知识点、错题本、AI 会话和社区内容等对象的业务数据模型。测试结果表明，系统已具备较完整的编程学习支撑能力，核心页面、主要接口和基础业务流程能够正常运行，但在导航交互、前端细节和部分扩展功能方面仍有进一步优化空间。

本文的工作重点在于面向编程学习场景构建一个多模块融合的综合教学辅助平台，并将在线判题安全隔离、AI 辅导接入和学习支持功能整合到统一系统中。该系统对编程教学平台的工程化实现和功能一体化设计具有一定参考价值。

**关键词：** 在线编程；自动判题；AI 答疑；学习路径；Spring Boot

## Abstract

With the rapid development of programming education, online programming learning platforms have become important tools in both classroom teaching and self-directed learning. Traditional platforms usually provide problem practice and basic judging functions, but they still have limitations in personalized learning support, instant tutoring, learning process tracking, and integrated teaching assistance. To address these issues, this thesis designs and implements a personalized online programming and tutoring system, aiming to provide learners with a comprehensive platform integrating online coding, automatic judging, AI-assisted tutoring, learning path management, wrong-question review, community interaction, and administrative management.

The system adopts a front-end and back-end separated architecture. The front end is built with Vue 3 and Vite, combined with Element Plus and Monaco Editor to provide an interactive interface and online coding environment. The back end is implemented with Spring Boot, MyBatis is used for data access, MySQL is used for data persistence, and JWT is applied for identity authentication and permission control. In the code execution and judging scenario, Docker containers are introduced to isolate user code execution, improving system security and operational stability. In the intelligent tutoring scenario, the system integrates a large language model API to support programming question answering, multi-turn conversations, code optimization suggestions, and knowledge explanation.

Centered on the core business requirements, the system implements modules for user and permission management, problem management, online code execution and judging, learning center and learning path management, wrong-question review and submission records, AI-based tutoring, community communication, and back-end administration. At the database level, a business data model is built to cover users, problems, test cases, submissions, learning paths, knowledge points, wrong-question records, AI sessions, and community content. Test results show that the system has achieved relatively complete support for programming learning, and its core pages, major APIs, and basic business workflows can operate normally. However, there is still room for improvement in navigation interaction, front-end details, and some extended functions.

The main contribution of this thesis lies in building an integrated teaching-support platform for programming learning scenarios, and in unifying secure online judging, AI tutoring, and learning support capabilities within one system. The system provides a practical reference for the engineering implementation and integrated design of programming education platforms.

**Keywords:** online programming; automatic judging; AI tutoring; learning path; Spring Boot

## 第1章 绪论

### 1.1 研究背景

近年来，随着软件产业持续发展以及高校信息类课程改革不断深入，编程能力培养已经成为计算机及相关专业人才培养中的重要内容。与此同时，在线教育平台和数字化教学工具在教学活动中的作用日益突出，在线编程学习、远程实验、自动测评和学习数据分析等应用场景不断增多。传统线下编程教学往往依赖本地开发环境配置、教师批改作业和课后答疑，存在环境部署复杂、反馈周期较长、学习过程难以量化等问题。

现有部分在线编程平台虽然提供了题目练习与自动判题能力，但大多聚焦于刷题和结果判定，难以覆盖学习者从入门到提升的完整学习过程。一方面，学习者在代码运行和提交失败后，往往只能获得有限的错误信息，缺少面向知识点和思路层面的指导；另一方面，平台对用户学习进度、知识掌握情况、错题复习路径以及学习过程数据的支持仍然不足。此外，随着大语言模型技术的发展，AI 在编程辅导、错误分析与知识讲解等方面展现出较强潜力，为构建智能化编程学习平台提供了新的技术基础。

基于此，设计一个面向编程学习场景的综合性平台具有较强的现实意义。该平台不仅需要具备题库管理、在线运行和自动判题等基础能力，还需要在用户学习支持、答疑辅导、过程记录和系统管理等方面进行综合设计，从而为学习者提供更加完整的学习闭环。

### 1.2 研究目的与意义

本文围绕“个性化在线编程与答疑系统”的设计与实现展开研究，其主要目的在于构建一个集在线编程、自动判题、AI 辅导、学习支持与后台管理于一体的教学辅助平台，提升编程学习过程中的即时反馈能力和平台综合服务能力。系统面向普通学习者和管理员两类主要角色，既关注用户完成题目练习、提交代码、复盘错题和查看学习进度等学习需求，也考虑题库维护、用户管理、权限分配和系统运营等管理需求。

从实践意义来看，本系统能够减少学习者在环境配置和结果验证上的额外成本，使其更加专注于编程思维训练和问题解决过程。通过在线运行、自动判题和 AI 辅导的结合，系统能够在学习者遇到代码错误、逻辑问题或知识盲点时提供更加及时的支持。通过学习路径、错题本、提交记录和知识图谱等功能，系统还能帮助学习者更好地理解自身学习状态，为后续学习安排提供依据。

从工程实现意义来看，本文将前后端分离架构、JWT 认证、Docker 容器隔离、AI 接口集成和分层后端结构整合到统一平台中，形成了一个较为完整的项目实现案例。该系统既具备一定的教学应用价值，也为在线编程学习平台的架构设计与功能融合提供了可参考的工程实践。

### 1.3 同类平台现状分析

目前常见的在线编程平台大体可以分为三类。第一类是以题目练习和自动判题为核心的在线评测平台，这类平台在题目组织、提交评测和排行榜等方面较为成熟，但通常更强调竞赛训练或刷题效率，对学习过程支持相对有限。第二类是以课程教学和内容学习为主的在线教育平台，这类平台重视课程资源和教学内容呈现，但在代码实时运行、判题反馈和针对性训练方面往往不够深入。第三类是加入智能问答能力的新型学习平台，这类平台在答疑效率上具有优势，但很多系统尚未将 AI 辅导与题目、提交记录、学习路径和错题复习等功能形成真正的一体化联动。

与普通 OJ 平台相比，编程学习平台在功能设计上更加关注“学”的过程，而不仅是“做题”的结果。对于初学者而言，仅有通过或未通过的判题结果并不足以支撑持续学习，他们更需要题目相关背景、错误定位、知识讲解和学习建议等信息。对于教学管理者而言，单纯的提交记录也不足以支撑教学分析，还需要从用户活跃度、学习路径完成情况、知识点分布、错题情况和后台数据管理等角度进行综合支持。

因此，构建一个融合题库、判题、AI 答疑、学习中心、错题复习、社区交流和后台管理的综合平台，是对传统在线编程系统的重要补充。本文所实现的系统正是围绕这一思路展开设计，强调系统功能的一体化和学习场景的连续性。

### 1.4 本文主要工作

本文的主要工作包括以下几个方面。

第一，完成了系统需求分析与总体方案设计，明确了用户端和管理端的核心业务需求，并基于前后端分离思想设计了系统总体架构、功能模块结构和数据库模型。

第二，完成了系统核心功能的实现。后端基于 Spring Boot 构建业务服务，前端基于 Vue 3 搭建用户界面，实现了用户认证、题目管理、代码运行、自动判题、AI 编程答疑、学习路径展示、错题本、提交记录、社区交流和后台管理等功能。

第三，在代码运行与判题场景中引入 Docker 容器隔离技术，实现了用户代码的受限执行和自动清理机制，提高了系统安全性；在智能辅导场景中接入大语言模型接口，实现了面向编程问题的问答与会话管理。

第四，围绕系统当前实现状态整理了数据库结构、接口文档和测试结果，对系统功能可用性进行了阶段性验证，并总结了目前存在的问题与后续优化方向。

## 第2章 相关技术与开发环境

### 2.1 前端相关技术

本系统前端采用 Vue 3 作为核心开发框架。Vue 3 具备良好的组件化开发能力和响应式机制，能够较好地满足中大型单页应用的开发需求。在本系统中，首页、题库、在线运行、AI 答疑、学习中心、错题本、提交记录、个人中心、社区和后台管理等页面均以组件化方式组织，便于复用与维护。

项目构建工具使用 Vite。相较于传统构建工具，Vite 在开发阶段具备更快的启动速度和热更新体验，适合前后端分离项目的持续迭代。前端依赖中还引入了 Vue Router 和 Pinia，前者用于组织页面路由，后者用于进行用户状态和主题等全局状态管理，从而实现多页面场景下的数据共享与访问控制。

在界面层面，系统使用 Element Plus 作为 UI 组件库，以提升界面开发效率和交互一致性。在线编程部分采用 Monaco Editor 作为代码编辑器组件，该编辑器能够提供语法高亮、代码编辑和更接近本地 IDE 的交互体验，有助于提升在线编程场景下的可用性。

### 2.2 后端相关技术

系统后端采用 Spring Boot 3.2.0 作为核心开发框架。Spring Boot 具有配置简化、依赖整合便利、开发效率高等特点，适合快速构建分层清晰的 Web 应用。在本系统中，后端按照控制层、业务层、数据访问层和实体层进行组织，便于不同功能模块的解耦与扩展。

数据访问层采用 MyBatis 实现。相比全自动 ORM 框架，MyBatis 更适合对 SQL 进行显式控制，便于编写题目查询、提交记录统计、学习路径关联等较复杂的业务查询。系统中各业务模块均通过 Mapper 接口与 XML 映射文件完成数据库交互，从而保持业务逻辑与数据访问逻辑的分离。

在认证与安全方面，系统引入 Spring Security 相关能力，并结合 JWT 实现前后端分离场景下的无状态认证。后端通过生成和校验 Token 来识别用户身份，并根据角色信息区分普通用户与管理员权限。除此之外，系统还使用 OkHttp 作为 HTTP 客户端，用于调用外部 AI 接口；使用 WebSocket 相关配置为实时交互场景预留支持；使用 Redis 相关依赖为缓存和扩展能力提供基础。

### 2.3 判题与隔离运行相关技术

在线编程平台与普通信息管理系统的显著差异之一在于需要运行用户提交的代码。若直接在服务端宿主环境执行用户代码，将面临系统资源被恶意占用、非法访问宿主机资源、死循环导致服务阻塞等风险。因此，系统引入 Docker 容器技术作为代码运行环境，为不同语言的代码执行提供隔离空间。

在具体实现上，系统通过 docker-java 客户端与 Docker 环境通信，根据代码语言选择不同镜像，例如 Java 使用 `openjdk:11-jdk-slim` 镜像，Python 使用 `python:3.9-slim` 镜像。在运行时为容器设置内存限制，并使用 `networkMode("none")` 关闭网络访问，从而减少代码执行对宿主环境的影响。代码运行结束后，系统会主动删除容器，降低资源泄露风险。对于批量测试用例的执行，系统使用线程池并发处理多个输入，提高评测效率。

### 2.4 数据存储与开发环境

系统采用 MySQL 作为主要关系型数据库，用于存储用户信息、题目数据、测试用例、提交记录、学习路径、知识点、错题本、AI 会话和社区内容等业务数据。根据现有数据库报告，系统数据库中已包含 50 张数据表，形成了较为完整的业务数据支撑体系。

开发环境方面，后端基于 JDK 17 和 Maven 构建，前端基于 Node.js 环境下的 npm 进行依赖管理和打包。系统在本地开发时采用前后端分离运行方式，前端开发端口为 3000，后端服务端口为 8080。系统的运行还依赖 Docker 环境以及可选的 Redis 服务。整体开发环境配置清晰，便于本地调试与后续部署。

## 第3章 系统需求分析

### 3.1 系统建设目标

本系统的建设目标并不仅仅是实现一个基础在线评测平台，而是面向编程学习场景构建一个综合性教学辅助系统。系统需要在完成题目练习、代码运行和自动判题等基础能力的同时，进一步支持学习过程管理、错题复盘、智能答疑和后台运营管理，从而形成更加完整的学习闭环。

具体而言，系统建设目标主要体现在以下几个方面。其一，为学习者提供安全、便捷的在线编程与提交环境，减少本地环境配置带来的学习门槛。其二，通过自动判题和详细结果反馈，提高学习者对代码正确性和问题定位的把握能力。其三，结合 AI 技术实现编程问答与知识讲解，增强系统的智能辅导能力。其四，通过学习路径、知识掌握度、错题本和提交记录等模块，强化平台对学习过程的支撑能力。其五，为管理员提供题库维护、用户管理和后台统计能力，保证平台长期可运营。

### 3.2 用户角色分析

系统当前主要面向两类角色展开设计，即普通用户和管理员。普通用户是系统的主要使用对象，其核心操作包括注册登录、浏览题目、在线编写代码、运行代码、提交判题、查看提交记录、使用 AI 进行编程答疑、查看学习路径与学习报告、管理错题本、浏览社区和查看个人中心等。管理员则负责对平台内容和用户进行管理，其核心操作包括用户状态管理、角色管理、题目维护、题目批量导入、知识点与学习路径维护、查看提交记录、审计日志和系统状态等。

从角色边界来看，普通用户主要聚焦于学习和使用场景，管理员主要聚焦于平台运营与内容管理场景。通过 JWT 中携带的角色信息以及后端拦截器校验机制，系统能够在接口访问层面对不同角色进行区分，从而实现基本的权限控制。

### 3.3 功能需求分析

结合当前项目实现情况，系统功能需求可以划分为以下几个方面。

第一，用户与权限管理需求。系统需要提供注册、登录、获取用户信息、更新用户资料和角色权限控制等能力，使不同身份的用户能够按照既定权限使用系统。

第二，题库与题目管理需求。系统需要支持题目列表查询、题目详情查看、题目测试用例管理、按标签或语言筛选题目，以及后台题目增删改查和批量导入等能力。

第三，在线代码运行与自动判题需求。系统需要允许用户在浏览器中直接编写并运行代码，系统能够根据题目测试用例对代码进行评测，并返回输出结果、错误信息或判题状态。

第四，学习支持需求。系统需要提供学习中心、学习路径、章节与关卡展示、知识图谱、知识掌握度、周报月报、薄弱项分析和个性化推荐等能力。需要说明的是，当前这部分功能已形成基础页面、接口和数据结构，但更强的动态个性化能力仍处于持续完善过程中。

第五，学习复盘与过程记录需求。系统需要记录用户提交历史，并自动形成错题本，为后续复习和查漏补缺提供依据，同时支持参考题解、提示和学习建议等辅助能力。

第六，社区与互动需求。系统需要提供发帖、评论、点赞、热点话题等功能，增强学习者之间的互动交流。

第七，后台管理需求。系统需要面向管理员提供用户管理、权限分配、班级管理、题目管理、知识点管理、学习路径管理、统计查看与系统监控等接口和页面，以保证平台具备持续运营能力。

### 3.4 非功能需求分析

除功能需求之外，系统还需要满足若干非功能需求。首先是安全性需求。在线运行用户代码本身具有较高风险，因此必须通过容器隔离、资源限制和超时控制等机制降低风险。用户认证信息也需要通过 Token 机制进行校验，并避免敏感信息泄露。

其次是可维护性需求。由于系统涉及前端页面、后端服务、数据库和外部 AI 接口等多类组件，必须采用较清晰的模块划分和分层设计，降低后续修改与扩展的复杂度。前后端分离架构和后端分层结构正是围绕这一需求展开。

再次是可扩展性需求。系统当前已经实现了学习路径、知识图谱、AI 对话和后台统计等基础能力，但后续仍可能引入更多语言支持、调试能力、更复杂的个性化分析和更完整的数据报表。因此，系统在模块设计上需要具备一定扩展空间。

最后是可用性需求。平台面向学习者使用，界面需要尽可能简洁清晰，功能流程需要易于理解。现有测试结果表明系统核心流程已经具备基础可用性，但前端导航一致性、交互细节和部分页面完整性仍需持续优化。

## 第4章 系统总体设计

### 4.1 系统总体架构设计

本系统采用典型的前后端分离架构。前端负责页面展示、用户交互、状态管理和接口调用；后端负责业务处理、权限校验、数据访问、代码执行调度和外部服务调用；MySQL 负责业务数据存储；Docker 负责用户代码的受限执行环境；外部 AI 服务负责提供智能问答和知识讲解能力。整体架构既便于前后端独立开发，也有助于提高系统维护效率和模块清晰度。

在用户访问过程中，前端首先通过路由进入不同功能页面，如题库页、学习中心页、AI 答疑页和后台管理页等。当前端需要获取数据时，会通过统一封装的 API 请求模块向后端发起 HTTP 请求。后端接收到请求后，根据接口归属交由相应控制器处理，再由 Service 层完成业务逻辑，并通过 Mapper 层访问数据库。当涉及在线运行和判题时，后端会调用代码沙箱服务，并进一步利用 Docker 工具类完成容器创建、代码执行和结果回收。

### 4.2 系统功能模块设计

从功能组织角度看，系统可以划分为用户模块、题库模块、代码运行与判题模块、提交与错题模块、AI 答疑模块、学习中心模块、社区模块、个人中心模块和后台管理模块。各模块既相对独立，又通过统一用户体系和数据模型建立联系。

从后端结构角度看，系统按照控制层、业务层、数据访问层和数据实体层进行设计。控制层用于暴露 RESTful 风格接口，对外提供功能入口；业务层负责封装具体业务逻辑；数据访问层通过 MyBatis Mapper 与数据库交互；实体层和 VO 层用于承载业务对象与接口返回对象。这种设计有利于将请求处理、业务逻辑和数据访问职责进行明确分离。

从前端结构角度看，系统基于视图层、接口层、状态管理层和公共组件层进行组织。视图层以页面为单位对应系统功能模块，接口层负责请求后端接口，状态管理层负责共享用户信息与主题状态，公共组件层则提供如代码编辑器、AI 对话框等复用能力。

### 4.3 业务流程设计

在普通用户使用场景中，典型业务流程可以概括为“登录系统、浏览题目、在线编码、运行代码、提交判题、查看结果、复盘错题、继续学习”。学习者首先通过注册或登录获取系统身份，进入题库后选择题目进行练习。编写完成后，可以先执行在线运行获取基本输出，再进行正式提交。后端接收提交请求后，调度代码沙箱执行用户代码，并结合测试用例生成判题结果。如果提交失败，相关记录可以进入错题本，为后续复习与强化训练提供支撑。若学习者在解题过程中遇到困难，还可以通过 AI 答疑模块获取思路提示、代码解释和知识讲解。

在管理员使用场景中，典型流程包括“登录后台、维护用户、维护题目、配置知识点和学习路径、查看平台统计与审计记录”。管理员通过后台接口完成平台基础数据和系统状态的维护，保证系统长期稳定运行。

### 4.4 数据库设计

数据库是支撑系统业务运行的核心基础。根据当前数据库报告，系统已构建较为完整的关系模型，共包含 50 张数据表。用户相关数据主要由 `user`、`role`、`user_role`、`permission` 等表支撑，用于记录用户身份、角色与权限信息。题目相关数据由 `problem`、`test_case`、`reference_solution` 等表支撑，用于记录题目内容、测试用例和参考题解。

提交与评测相关数据主要由 `submit` 和 `submit_test_case_result` 等表支撑，用于记录用户提交内容、判题结果与测试用例级结果。学习支持相关数据由 `learning_path`、`path_chapter`、`path_level`、`path_level_problem`、`knowledge_point`、`knowledge_module`、`learning_resource`、`learn_record` 和 `knowledge_mastery` 等表组成，用于支撑学习路径组织、知识点关联和学习过程记录。错题复习相关功能由 `wrong_book` 与 `wrong_book_item` 等表支撑，AI 问答和会话相关能力由 `question`、`ai_session`、`ai_message` 和 `ai_collection` 等表支撑，社区与个人中心相关功能则对应 `community_post`、`community_comment`、`favorite` 等表。

这种数据库设计能够较好地支撑系统多模块协同运行，并为后续学习分析和平台扩展提供数据基础。

## 第5章 系统详细设计与实现

### 5.1 用户模块设计与实现

用户模块是系统运行的基础。系统通过 `UserController` 提供用户注册、登录、信息获取和语言偏好更新等接口。用户登录成功后，后端通过 `JwtUtil` 生成包含用户 ID 和角色信息的 Token，并返回给前端保存。之后前端在访问需要鉴权的接口时，将 Token 放入请求头中，由后端拦截器解析和校验，从而实现无状态认证。

在权限控制方面，系统通过普通用户与管理员两种角色区分不同功能访问范围。后端的 `JwtInterceptor` 负责对普通鉴权接口进行用户身份校验，`AdminInterceptor` 则负责对管理类接口进行更严格的角色判断。该机制能够满足系统现阶段对权限管理的基本要求。

### 5.2 题库与题目管理模块设计与实现

题库模块是平台学习内容组织的核心。系统通过 `ProblemController` 提供题目列表查询、题目详情查看、样例测试用例获取、完整测试用例获取以及按标签查询等接口。题目信息包括标题、内容、输入输出说明、难度、语言、标签、知识点、提示信息和样例解释等字段，可以较好地支撑用户浏览和练习需求。

管理端通过 `AdminController` 和 `FileController` 实现题目维护与批量导入能力。管理员可以新增、修改、删除题目，也可以通过 Excel 或 CSV 文件导入题目数据，从而提高题库维护效率。这种设计使平台既适用于个人学习场景，也具备一定教学内容管理能力。

### 5.3 在线代码运行与自动判题模块设计与实现

在线代码运行与自动判题模块是系统最核心的功能之一。系统通过 `CodeController` 提供在线运行接口，通过 `SubmitController` 提供正式提交与结果查询接口。在用户点击运行代码时，系统主要返回程序输出和错误信息，用于帮助用户进行快速调试；在用户进行正式提交时，系统则结合题目测试用例进行完整判题，并记录提交结果。

在具体实现上，`CodeSandboxServiceImpl` 负责代码执行调度，其核心实现是调用 `DockerUtil` 在 Docker 容器中执行用户代码。系统根据语言类型选择不同运行镜像，并构造相应命令，将代码写入容器环境中进行编译或解释执行。为了降低风险，系统为容器设置内存限制并关闭网络访问。执行结束后，系统收集标准输出和标准错误信息，根据输出前缀和退出状态判断执行结果，并在最终阶段强制删除容器，避免容器资源残留。

在批量判题场景中，系统通过线程池并发执行多个测试输入，并将执行结果封装为 `CodeExecutionResult` 对象列表。该方式能够在一定程度上提升评测效率，同时将编译错误、运行错误和一般执行异常进行分类处理，便于后续展示给用户。

### 5.4 AI 智能答疑模块设计与实现

AI 智能答疑模块是本系统相较于传统在线评测平台的重要扩展。系统通过 `AiController` 提供普通提问、多轮聊天、会话历史查询、错误诊断、代码优化建议和学习建议等接口。用户既可以针对某一编程问题进行单次提问，也可以在已有会话中进行连续追问，从而增强答疑的连贯性。

在实现层面，`AiServiceImpl` 负责 AI 调用和会话数据管理。系统从请求中获取问题描述和可选代码片段，将其与预设提示词拼接后，借助 OkHttp 向外部大模型接口发起请求，并解析返回结果。对于多轮对话场景，系统会先查询历史消息，将历史消息和当前用户消息一并发送给模型，从而保留上下文语义；随后将用户消息和模型回复一起写入数据库中的会话记录表，便于后续查看历史内容。除此之外，系统还提供了问答收藏等辅助功能，增强了 AI 模块的可用性。

### 5.5 学习中心与学习路径模块设计与实现

学习中心模块体现了本系统区别于单纯刷题平台的学习支持能力。系统通过 `LearnController` 提供学习统计、推荐题目、能力测评、学习路径、章节详情、关卡解锁、知识图谱、知识掌握度、周报月报和薄弱项分析等接口。前端则通过学习中心、学习路径详情和关卡详情等页面将相关数据展示给用户。

从数据组织方式来看，系统围绕学习路径、章节、关卡和题目建立了关联结构，并通过知识点和知识模块扩展学习内容组织能力。当前数据库中已经包含学习路径、章节、关卡、关卡题目关联、知识点和知识模块等数据表，能够支撑基础的学习路径展示与学习统计功能。根据现有数据库报告，系统当前包含 6 条学习路径、36 个章节和 93 个关卡，说明学习内容组织能力已经具备一定规模。

需要说明的是，学习路径模块当前已经具备路径展示、基础测评、章节进度和部分学习报告能力，但更细粒度的动态个性化调整、更加完善的知识图谱分析以及更强的自适应推荐能力仍在持续完善中。因此，在论文表述中应将当前状态界定为“已完成基础实现并具备扩展能力”，而不是夸大为完全成熟的个性化推荐系统。

### 5.6 错题本、提交记录与参考题解模块设计与实现

为了帮助学习者进行问题复盘，系统设计了提交记录、错题本和参考题解三类辅助功能。提交记录模块通过 `SubmitController` 提供用户提交历史查询和提交详情查看能力，便于学习者回顾历史代码和评测结果。错题本模块通过 `WrongBookController` 提供错题列表、错题详情、复习计划、待复习错题、统计信息以及基于错题的推荐题目等接口，有助于学习者围绕薄弱点进行强化训练。

此外，系统还通过 `ReferenceSolutionController` 提供参考题解和提示信息接口，使用户在完成题目练习后能够进一步理解题目思路和标准解法。提交记录、错题本和参考题解共同构成了学习复盘闭环，有助于提升系统对学习过程的支持深度。

### 5.7 社区、个人中心与后台管理模块设计与实现

在社区模块中，系统通过 `CommunityController` 提供帖子浏览、帖子详情、发帖、评论、点赞、用户帖子与热点话题等接口，用于支持学习者之间的交流互动。虽然社区功能并不是系统最核心的教学模块，但它能够增强平台的参与感和持续使用动力。

个人中心模块通过 `UserProfileController` 提供用户资料、统计信息、活动记录、成就信息和收藏信息等接口，用于展示用户个人学习概况和平台内行为数据。这一模块有助于用户从更宏观的视角理解自身的学习状态。

后台管理模块通过 `AdminController` 提供用户管理、角色与权限管理、题目管理、知识点管理、学习路径管理、班级管理、审计日志、系统指标和仪表盘等接口。该模块使系统不再只是单一的学习工具，而具备一定的平台运营和管理能力。

### 5.8 前端页面与交互实现

前端部分按照功能页面进行组织，主要页面包括首页、题库、题目详情、在线运行、AI 答疑、学习中心、学习路径、错题本、提交记录、社区、个人中心和后台管理等。页面之间通过 Vue Router 进行路由切换，并通过统一的 API 模块向后端发起请求。对于用户信息和主题等公共状态，系统通过 Pinia 进行统一管理。

从交互设计上看，系统尽量围绕学习场景进行页面布局。例如，题目详情页与代码编辑器联动，方便用户直接完成题目练习；学习中心页面汇集了学习路径、推荐内容、知识掌握度和学习报告等信息，便于用户查看整体学习状态；AI 对话、错题本和提交记录等页面则作为学习辅助页面，与做题和复盘过程形成配合。总体而言，前端实现已具备较为完整的功能框架，但仍需在部分导航刷新、页面一致性和细节安全性方面继续优化。

## 第6章 系统测试

### 6.1 测试环境与测试方法

系统测试主要围绕功能可用性进行。结合现有项目材料，测试工作包括前端页面测试、接口测试和基础业务流程验证等内容。前端测试主要通过基于 Playwright 的自动化方式检查页面导航、页面元素完整性和部分交互逻辑；接口测试则通过项目中的若干测试脚本和接口文档对注册登录、学习模块、后台接口等进行验证。

测试环境采用本地前后端分离部署方式，前端运行于 `http://localhost:3000`，后端运行于 `http://localhost:8080`。系统在测试过程中重点关注用户登录状态维持、页面访问、接口响应、判题流程、学习中心数据展示和后台接口可访问性等内容。

### 6.2 功能测试结果

从现有测试材料来看，系统已完成对多个核心页面和业务流程的阶段性验证。前端测试报告显示，学习中心、题库、学习社区、在线运行、AI 答疑、错题本、提交记录和个人主页等页面的基本元素已经完成检查，直接通过 URL 访问页面整体正常，API 请求能够正常发起，登录状态也能够维持。这说明系统在整体功能框架上已经基本成型。

数据库报告显示，系统已具备较为完整的数据支撑能力，当前数据库中包括 90 道题目、6 条学习路径、36 个章节、93 个关卡、39 个知识点和 218 个测试用例。这些数据说明系统并非停留在空壳界面阶段，而是已经具备一定的内容规模和业务数据积累。

此外，后端控制器和接口文档表明，系统已实现用户管理、题目管理、代码运行、提交记录、AI 会话、学习路径、错题本、社区管理和后台管理等多类接口，功能覆盖面较广，能够支撑论文中“综合性在线编程学习平台”的定位。

### 6.3 测试中发现的问题与改进方向

尽管系统已经具备较完整的功能框架，但现阶段测试也暴露出一些问题。根据前端测试报告，在页面导航过程中存在“URL 正确变化但页面内容未及时刷新”的问题，这会直接影响用户在不同页面之间切换时的使用体验。此外，测试还发现控制台中存在输出完整 JWT Token 的情况，这在安全性上存在明显风险，应尽快移除敏感日志输出。部分页面在导航切换后元素显示不完整，也说明前端在页面渲染和状态同步方面仍需进一步优化。

这些问题表明，系统目前已经具备中期阶段的核心能力，但在前端交互稳定性、细节安全性和部分页面完整性方面还未达到最终版本要求。后续优化工作应重点围绕路由刷新机制、日志安全、页面状态同步和用户体验细节展开。

### 6.4 测试结论

总体来看，系统已经完成了在线编程学习平台所需的主要功能模块搭建，能够支撑用户完成从登录、做题、运行、提交、查看结果到使用学习辅助功能的基本流程。数据库、接口与前端页面之间已经形成较完整的协同关系，说明系统总体方案具备可行性。

与此同时，测试结果也反映出系统仍处于持续完善阶段，尤其在前端稳定性和部分扩展功能上还有进一步优化空间。因此，可以将当前系统评价为“已完成阶段性核心目标，具备继续完善和交付的基础”，这一结论也与毕业设计中期阶段的项目状态基本一致。

### 7.1 工作总结

本文围绕个性化在线编程与答疑系统的设计与实现，完成了系统需求分析、总体方案设计、数据库结构组织以及多个核心模块的开发工作。系统采用 Vue 3 与 Spring Boot 构建前后端分离平台，结合 MyBatis、MySQL、JWT、Docker 和 AI 接口等技术，实现了用户管理、题库管理、在线代码运行、自动判题、AI 答疑、学习路径、错题本、社区交流和后台管理等功能。

与仅提供刷题和判题能力的普通平台相比，本系统更加重视编程学习过程中的支持能力。通过学习中心、错题本、提交记录、参考题解和 AI 辅导等模块，系统尝试在“学、练、评、答、复盘”之间建立联系，形成更加完整的学习辅助闭环。与此同时，系统在工程实现上也体现了分层设计、前后端解耦和容器隔离执行等较为典型的工程实践思路。

### 7.2 系统不足

虽然系统已具备较为完整的功能框架，但仍然存在一些不足。第一，学习路径和知识图谱相关功能已经形成基础页面和接口，但真正意义上的动态个性化推荐和更深入的学习分析能力仍需进一步加强。第二，前端交互在路由刷新、页面一致性和细节安全性方面仍存在问题，需要继续打磨。第三，当前系统支持的编程语言仍较为有限，在线调试和更复杂的判题能力也尚有拓展空间。第四，系统测试以功能验证为主，后续仍需开展更加系统的性能测试、安全测试和长期稳定性验证。

### 7.3 后续展望

后续工作可以从以下几个方向展开。第一，继续完善学习路径与知识图谱模块，增强基于学习过程数据的分析和推荐能力，使系统在“个性化”方面体现出更强的实际效果。第二，优化 AI 辅导模块，使其与当前题目、提交记录和知识点数据进行更深入联动，提升答疑的上下文感知和针对性。第三，进一步扩展在线编程能力，增加更多编程语言支持，并探索更丰富的调试和评测方式。第四，完善后台统计、教学运营和系统监控能力，为后续教学应用和平台运营提供更强支撑。第五，结合学校论文规范继续补充图表、参考文献和格式化内容，使论文和系统交付成果更加完整。
