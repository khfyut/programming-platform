# 仓库整理说明

本次整理的目标是把仓库从“开发过程堆积目录”收敛成“可持续维护的项目目录”。

## 已完成的整理

### 1. 根目录收缩

根目录现在只保留：

- 正式代码目录：`frontend/`、`System/`
- 正式数据目录：`db/`
- 脚本目录：`scripts/`
- 测试目录：`tests/`
- 压测目录：`loadtest/`
- 文档目录：`docs/`
- 入口文件：`README.md`、`.gitignore`、`.env.example`、`LICENSE`

### 2. 文档归位

原先散落在根目录的设计、报告、论文材料已经分别移动到：

- `docs/design/`
- `docs/reports/`
- `docs/testing/`
- `docs/thesis/`
- `docs/archive/`

### 3. 数据脚本归位

当前仍有效的数据库补丁保留在：

- `db/seed/`

历史 SQL、初始化脚本、一次性插入脚本已归档到：

- `db/archive/legacy/`
- `db/archive/backups/`

### 4. 测试资产归位

手工测试脚本移动到：

- `tests/manual/`

测试截图移动到：

- `docs/testing/screenshots/`

压测摘要移动到：

- `loadtest/results/`

### 5. 已清理的噪音

已删除：

- 根目录 0 字节空脚本和空报告
- 浏览器自动生成的 `chrome-profile`
- 本地 `.m2/`
- 本地 `.trae/`
- `System/loadtest-run-output` 中的大量原始 HTML/JTL 报告
- 压测运行过程中生成的后端 / JMeter 临时日志

## 当前保留策略

### 应保留

- 正式前后端代码
- 正式数据库补丁
- 仍然有复用价值的脚本
- 最终设计文档、测试报告、论文材料

### 应归档

- 旧版 SQL
- 一次性过程方案
- 历史系统记录
- 人工测试截图和过程材料

### 不建议再放进仓库根目录

- 临时 Python 脚本
- 一次性 SQL
- 临时 Markdown 方案
- 浏览器缓存或测试运行时 profile
- 本地 IDE / Maven 缓存

## 还可以继续做的收尾

- 统一补 `docs/api.md`、`docs/deployment.md`、`docs/acceptance.md`
- 给 `tests/` 再拆 `api/`、`e2e/`
- 给 `db/` 再拆 `migration/`
- 如果决定彻底分离论文材料，可把 `docs/thesis/` 单独迁出到独立仓库
