# 交付前 Review 报告

## 1. 审查目标

本次审查面向项目交付阶段，重点确认以下内容是否达到可交付状态：

- 前端代码是否可构建、主要页面是否具备交付条件
- 后端代码是否可编译、可启动，关键接口是否存在明显运行期风险
- 数据库结构、核心业务数据、历史报告与当前真实数据是否一致
- 是否存在会影响答辩、演示、验收或上线的高风险问题

审查时间：2026-04-02

## 2. 审查方式

- 静态审查前端、后端核心代码与 Mapper/Controller/Service 调用链
- 执行前端构建与 lint
- 执行后端编译与 Spring Boot 启动验证
- 对 MySQL 进行只读查询，核对核心表结构、数据规模与完整性

## 3. 总体结论

当前项目 **不建议直接进入最终交付**。

原因不是“项目完全不可用”，而是目前同时存在以下几类交付阻塞问题：

1. 部分核心功能存在运行期故障风险，尤其是个人主页相关接口
2. 多个接口存在基于 ID 的越权读取/修改风险
3. 前后端有真实的接口对接错位，部分页面会展示错误数据或直接 404
4. 学习分析与推荐中存在明显的占位/硬编码逻辑，不适合直接作为成品能力对外说明
5. 数据库当前真实数据与仓库内已有数据库报告明显不一致，且存在孤儿数据与空数据域

## 5. 主要问题

### P0-1 个人主页相关功能存在运行期 MyBatis 绑定失败风险

## 7. 其他观察

### 7.1 前端包体偏大

虽然 `vite.config.js` 已做 `manualChunks`，但构建结果中仍出现超大 chunk 警告：

- `monaco` 包体约 `2.1MB`
- `element-plus` 包体约 `799KB`

这不是当前交付阻塞项，但会影响首屏加载与演示体验。

### 7.2 业务数据偏稀疏

- 社区仅 `2` 篇帖子、`0` 条评论
- AI 仅 `1` 个会话、`2` 条消息
- 提交仅 `23` 条

如果交付目标包含“展示活跃度”和“验证多模块联动”，现有演示数据偏少。

## 8. 建议整改顺序

### 第一优先级（必须完成后再交付）

1. 补齐 `CommunityMapper.xml` 与 `SubmitMapper.xml` 中缺失的映射语句
2. 为提交详情、错题本、AI 会话接口增加“资源归属校验”
3. 修正 `UserProfile.vue` 的数据来源，明确“当前用户页”和“公开用户页”两套接口
4. 修正 `Collections.vue` 中不存在的 `/api/community/my/posts` 调用
5. 清理孤儿测试用例，更新数据库报告为当前真实数据

### 第二优先级（建议在答辩/验收前完成）

1. 清理学习分析中的硬编码与示例数据回退
2. 回填 `user_knowledge_mastery`、`review_plan`、社区与 AI 示例数据
3. 补充至少一轮接口级联调验证证据
4. 降低前端首包体积，尤其是 Monaco 和 Element Plus 相关 chunk

## 9. 最终判断

当前版本具备“继续整改后完成交付”的基础，但 **还不适合直接作为最终交付版提交**。

如果现在就进入交付，最容易在现场暴露的问题是：

- 个人主页相关模块直接报错
- 登录用户能看到或修改别人的提交/错题/AI 会话
- 公开主页展示的数据归属混乱
- 数据库报告与现场数据库实际数据对不上
- 学习分析内容被追问后无法自洽

建议在完成“第一优先级”整改后，再做一次针对前后端联调与数据库的一轮复审。
