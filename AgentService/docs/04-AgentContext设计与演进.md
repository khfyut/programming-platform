# 学习笔记 04：AgentContext设计与演进

## 1. 核心概念
**本质**：AgentContext的演进与分层设计

**从简单到复杂的过程**：
- 第一步：平铺字段（快速验证）
- 第二步：按领域拆分（结构清晰）
- 第三步：补全业务维度（支撑决策）

**知识类型**：🟢 Agent系统特有知识

## 2. 设计动机
**为什么要分层设计AgentContext？**

| 阶段 | 问题 | 解决 |
|------|------|------|
| 平铺 | 字段混乱，职责不清 | 按领域拆分子模型 |
| 拆分 | 信息不足，无法做复杂决策 | 补全LearnerState、SessionContext |

**反例**：一直保持平铺结构
- 50个字段的类，无法维护
- 不知道哪些字段属于哪个领域
- Java组装时容易遗漏

## 3. 关键结构

```python
AgentContext (整体上下文)
├── ProblemContext (题目事实)
├── SubmissionContext (提交事实)
├── LearnerState (学习者状态事实)
├── SessionContext (会话推断状态)
└── 其他元信息
```

**设计原则**：
- 子模型按**领域**划分
- 子模型内部按**使用场景**组织字段
- 用嵌套表达**包含关系**

## 4. 调用链路

```
Java查询多个表
    ↓
组装成多个子模型
    ↓
组合成AgentContext
    ↓
JSON序列化 → HTTP发送
    ↓
Python反序列化成对象
    ↓
各层按需访问不同子模型
```

## 5. 常见错误

| 错误 | 原因 | 解决 |
|------|------|------|
| 循环导入 | 模型A引用B，B引用A | 用字符串延迟引用或合并文件 |
| 字段遗漏 | 新增需求没补模型 | 建立模型变更检查清单 |
| 命名不一致 | 有的用id，有的用_id | 统一命名规范 |

## 6. 可复用模板

**上下文模型设计模板**：
```python
class XxxContext(BaseModel):
    # 1. 元信息（必带）
    request_id: str
    timestamp: datetime

    # 2. 业务事实（可信状态）
    entity: EntityContext

    # 3. 用户状态（可信状态）
    user: UserState

    # 4. 会话状态（推断状态）
    session: SessionContext
```

## 8. 判断标准

**"什么时候该拆分子模型？"**
> 当字段超过10个，或明显属于不同领域（题目/提交/用户/会话）时。

**"新字段该放到哪个子模型？"**
> 看它的**数据来源**和**使用场景**，不是随意放。

**【面试高频点】**
**Q：如何设计一个AI系统的上下文模型？**
**A**：按领域分层设计：1）元信息层（请求ID、时间戳）；2）业务事实层（题目、提交记录，来自数据库）；3）用户状态层（掌握度、历史记录）；4）会话推断层（策略、意图，Agent维护）。每层职责清晰，便于Java组装和Agent使用。
