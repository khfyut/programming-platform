# 代码运行功能流程图

以下流程图基于当前项目中在线代码运行功能的最新实现生成，主要对应：

- `System/src/main/java/com/programming/controller/CodeController.java`
- `System/src/main/java/com/programming/service/impl/CodeSandboxServiceImpl.java`
- `System/src/main/java/com/programming/util/DockerUtil.java`

本版本已同步反映以下修复后的真实行为：

- 用户代码不再通过 `echo "..."` 写入容器，而是使用 `Base64` 解码落盘
- 输入数据会通过标准输入真正传给程序
- Java 代码会先编译再运行
- 错误会区分为“编译错误”和“运行错误”
- 题目内存限制会先做单位归一化，再传给 Docker

## 1. 在线代码运行主流程

```mermaid
flowchart TD
    A["用户在前端输入代码、语言和输入数据"] --> B["前端调用 POST /api/code/run"]
    B --> C["CodeController.runCode 接收请求"]
    C --> D["提取 code、language、input"]
    D --> E["调用 CodeSandboxService.runCode(code, language, input)"]
    E --> F["CodeSandboxServiceImpl 转发到 DockerUtil.runCode"]
    F --> G["DockerUtil 计算执行参数"]
    G --> H["初始化 Docker Client"]
    H --> I["根据语言选择镜像"]
    I --> J{"语言是否支持"}
    J -- "否" --> K["返回 不支持的编程语言"]
    J -- "是" --> L["构造容器执行命令"]
    L --> M["源码做 Base64 编码"]
    M --> N["在容器内 Base64 解码为源文件"]
    N --> O["按题目或系统配置设置时间和内存限制"]
    O --> P["创建 Docker 容器"]
    P --> Q["设置 networkMode = none"]
    Q --> R["启动容器"]
    R --> S["收集 stdout / stderr 日志"]
    S --> T["读取容器退出码"]
    T --> U{"退出码是否为 0"}
    U -- "是" --> V["返回程序标准输出"]
    U -- "否" --> W["按退出码映射编译错误/运行错误"]
    V --> X["强制删除容器"]
    W --> X
    X --> Y["CodeController 封装 ResultUtil 返回前端"]
```

## 2. Java 运行子流程

```mermaid
flowchart TD
    A["收到 Java 源码和输入数据"] --> B["源码 Base64 编码"]
    B --> C["容器内解码写入 Main.java"]
    C --> D["执行 javac Main.java"]
    D --> E{"编译是否成功"}
    E -- "否" --> F["读取 compile.err"]
    F --> G["返回 编译错误"]
    E -- "是" --> H{"是否存在输入数据"}
    H -- "是" --> I["输入 Base64 编码后通过标准输入传给 java Main"]
    H -- "否" --> J["直接执行 java Main"]
    I --> K["读取 stdout / stderr"]
    J --> K
    K --> L{"运行是否成功"}
    L -- "否" --> M["读取 runtime.err"]
    M --> N["返回 运行错误"]
    L -- "是" --> O["返回程序输出"]
```

## 3. Python 运行子流程

```mermaid
flowchart TD
    A["收到 Python 源码和输入数据"] --> B["源码 Base64 编码"]
    B --> C["容器内解码写入 main.py"]
    C --> D{"是否存在输入数据"}
    D -- "是" --> E["输入 Base64 编码后通过标准输入传给 python3 main.py"]
    D -- "否" --> F["直接执行 python3 main.py"]
    E --> G["读取 stdout / stderr"]
    F --> G
    G --> H{"运行是否成功"}
    H -- "否" --> I["返回 运行错误"]
    H -- "是" --> J["返回程序输出"]
```

## 4. 异常分类逻辑

```mermaid
flowchart TD
    A["容器执行完成"] --> B["读取 exitCode"]
    B --> C{"exitCode == 0"}
    C -- "是" --> D["运行成功，返回 stdout"]
    C -- "否" --> E{"exitCode == 21"}
    E -- "是" --> F["判定为编译错误"]
    E -- "否" --> G{"exitCode == 22"}
    G -- "是" --> H["判定为运行错误"]
    G -- "否" --> I["按通用运行失败处理"]
```

## 5. 论文中可直接配套说明的文字

可直接配合流程图使用以下表述：

“系统在线代码运行功能采用控制层、业务层与容器执行层协作的方式实现。当前端提交代码运行请求后，后端控制器接收代码、语言与输入参数，并调用代码沙箱服务。代码沙箱服务进一步通过 Docker 工具类完成镜像选择、执行参数计算与容器调度。为避免源代码中的转义字符在 Shell 环境中被错误处理，系统先对用户代码进行 Base64 编码，再在容器内部解码落盘为源文件。对于 Java 程序，系统先完成编译，再根据用户输入通过标准输入流执行主程序；对于 Python 程序，则直接在容器中解释执行。执行结束后，系统收集标准输出、标准错误和退出码，并区分编译错误、运行错误与成功输出，最终销毁容器并将结果统一返回前端页面。”
