## 二、代码编译错误

### 2.1 javax.annotation包不存在

**问题描述**：
```
[ERROR] 程序包javax.annotation不存在
[ERROR] 找不到符号: 类 Resource
```

**原因**：Spring Boot 3.x使用Jakarta EE，`javax.annotation.Resource`已被移除

**解决方案**：
```java
// 修改前
import javax.annotation.Resource;
@Resource
private CodeSandboxService codeSandboxService;

// 修改后
import org.springframework.beans.factory.annotation.Autowired;
@Autowired
private CodeSandboxService codeSandboxService;
```

**关键点**：
- Spring Boot 3.x推荐使用`@Autowired`而不是`@Resource`
- 或使用`jakarta.annotation.Resource`

---

### 2.2 CodeSandboxService方法不存在

**问题描述**：
```
[ERROR] 找不到符号: 方法 runCode(java.lang.String,java.lang.String,java.lang.String)
```

**原因**：Service接口和实现类缺少方法定义

**解决方案**：
```java
// CodeSandboxService.java
public interface CodeSandboxService {
    String runCode(String code, String language, String input);
}

// CodeSandboxServiceImpl.java
@Service
public class CodeSandboxServiceImpl implements CodeSandboxService {
    @Autowired
    private DockerUtil dockerUtil;

    @Override
    public String runCode(String code, String language, String input) {
        return dockerUtil.runCode(code, language, input);
    }
}
```

### 2.3 Docker Java API Duration类型错误

**问题描述**：
```
[ERROR] 无法将类 ApacheDockerHttpClient.Builder中的方法 connectionTimeout应用到给定类型;
需要: java.time.Duration
找到: int, java.util.concurrent.TimeUnit
```

**原因**：Apache HttpClient 5.x的`connectionTimeout()`方法需要`Duration`类型

**解决方案**：
```java
// 添加导入
import java.time.Duration;

// 修改前
.connectionTimeout(30, TimeUnit.SECONDS)
.responseTimeout(30, TimeUnit.SECONDS)

// 修改后
.connectionTimeout(Duration.ofSeconds(30))
.responseTimeout(Duration.ofSeconds(30))
```

---

### 2.4 PullResponseItem回调类型错误

**问题描述**：
```
[ERROR] 方法不会覆盖或实现超类型的方法
[ERROR] 无法将接口 AsyncDockerCmd中的方法 exec应用到给定类型;
需要: T
找到: <匿名com.github.dockerjava.api.model.PullResponseItem>
```

**原因**：直接实现`PullResponseItem`接口导致类型不匹配

**解决方案**：
```java
// 修改前
.exec(new PullResponseItem() {
    @Override
    public void onNext(PullResponseItem item) {
        log.info("拉取镜像 {} 进度：{}", imageName, item.getStatus());
    }
})

// 修改后
.exec(new com.github.dockerjava.api.async.ResultCallback.Adapter<PullResponseItem>() {
    @Override
    public void onNext(PullResponseItem item) {
        log.info("拉取镜像 {} 进度：{}", imageName, item.getStatus());
    }
})
```

### 2.5 Frame.StreamType枚举不存在

**问题描述**：
```
[ERROR] 找不到符号: 变量 StreamType
[ERROR] 位置: 类 com.github.dockerjava.api.model.Frame
```

**原因**：Docker Java API的`Frame`类在某些版本中不提供`StreamType`枚举

**解决方案**：
```java
// 修改前
if (frame.getStreamType() == com.github.dockerjava.api.model.Frame.StreamType.STDOUT) {
    output.append(new String(frame.getPayload(), StandardCharsets.UTF_8));
} else if (frame.getStreamType() == com.github.dockerjava.api.model.Frame.StreamType.STDERR) {
    error.append(new String(frame.getPayload(), StandardCharsets.UTF_8));
}

// 修改后
String payload = new String(frame.getPayload(), StandardCharsets.UTF_8);
if (frame.toString().contains("STDOUT")) {
    output.append(payload);
} else {
    error.append(payload);
}
```

## 三、应用启动错误

### 3.1 MyBatis-Plus与Spring Boot 3.x不兼容

**问题描述**：
```
java.lang.IllegalArgumentException: Invalid value type for attribute 'factoryBeanObjectType': java.lang.String
```

**原因**：MyBatis-Plus 3.5.5与Spring Boot 3.2.0不完全兼容

**解决方案**：

#### 方案1：升级MyBatis-Plus版本（尝试但失败）
```xml
<version>3.5.7</version>
```

#### 方案2：改用MyBatis原生（最终采用）
```xml
<!-- 移除MyBatis-Plus -->
<dependency>
    <groupId>org.mybatis.spring.boot</groupId>
    <artifactId>mybatis-spring-boot-starter</artifactId>
    <version>3.0.3</version>
</dependency>
```

**关键点**：
- Spring Boot 3.2.0与MyBatis原生3.0.3完全兼容
- MyBatis-Plus需要等待兼容Spring Boot 3.2.x的版本

### 3.2 Mapper Bean缺失

**问题描述**：
```
Field problemMapper in com.programming.service.impl.ProblemServiceImpl required a bean of type 'com.programming.mapper.ProblemMapper' that could not be found.
```

**原因**：MyBatis-Plus被注释后，Service实现类中的Mapper注入失效

**解决方案**：
```java
// 临时注释（当MyBatis-Plus被禁用时）
// @Autowired
// private ProblemMapper problemMapper;

// 恢复（当MyBatis启用时）
@Autowired
private ProblemMapper problemMapper;
```

## 四、Docker配置问题

### 4.1 Docker API连接测试

**问题描述**：
```
curl: (28) Failed to connect to 192.168.59.133 port 2375 after 21044 ms
```

**原因**：Docker远程API未启用或配置错误

**解决方案**：

#### 在虚拟机上配置Docker远程API
```bash
# 创建配置目录
sudo mkdir -p /etc/systemd/system/docker.service.d

# 创建覆盖配置
sudo bash -c 'cat > /etc/systemd/system/docker.service.d/override.conf << EOF
[Service]
ExecStart=
ExecStart=/usr/bin/dockerd -H fd:// -H tcp://0.0.0.0:2375
EOF'

# 重新加载配置
sudo systemctl daemon-reload

# 重启Docker
sudo systemctl restart docker

# 验证端口
sudo netstat -tlnp | grep 2375
```

#### 在Windows上测试连接
```bash
curl http://192.168.59.133:2375/version
```

**成功标志**：返回Docker版本JSON信息

### 4.3 Windows路径绑定到Linux容器失败

**问题描述**：
```
Status 500: {"message":"invalid volume specification: 'C:\\\\Users\\\\33328\\\\AppData\\\\Local\\\\Temp\\\\...:/app:rw'"}
```

**原因**：Windows本地路径无法直接绑定到Linux Docker容器

**解决方案**：

#### 方案1：使用相对路径（尝试但失败）
```java
.withBinds(new Bind(
    dir.getAbsolutePath(),
    new Volume("/app")
))
```

#### 方案2：改用容器内文件操作（最终采用）
```java
// 移除本地文件创建和绑定
// String tempDir = System.getProperty("java.io.tmpdir") + File.separator + UUID.randomUUID();
// File dir = new File(tempDir);

// 直接在容器内创建和执行文件
runCommand = String.format(
    "echo '%s' > /app/Main.java && javac /app/Main.java && echo '%s' | java -cp /app Main",
    code.replace("'", "\\'").replace("\n", "\\n"),
    input.replace("'", "\\'")
);
```

**优势**：
- 不依赖本地文件系统
- 避免跨平台路径问题
- 更安全，完全容器隔离

## 五、PowerShell curl语法问题

**问题描述**：
```
Invoke-WebRequest : 无法绑定参数"Headers"
无法将"System.String"类型的"Content-Type: application/json"值转换为"System.Collections.IDictionary"类型
```

**原因**：PowerShell的curl命令语法与bash不同

**解决方案**：

#### 方案1：使用Invoke-WebRequest
```powershell
$body = @{
    code = "..."
    language = "java"
    input = ""
}

$headers = @{
    "Content-Type" = "application/json"
}

$response = Invoke-WebRequest -Uri "http://localhost:8080/api/code/run" -Method POST -Body ($body | ConvertTo-Json) -Headers $headers
```

#### 方案2：使用Postman（推荐）
- 下载Postman
- 创建POST请求
- 设置URL、Headers、Body
- 点击Send

#### 方案3：使用浏览器开发者工具
```javascript
fetch('http://localhost:8080/api/code/run', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json'
    },
    body: JSON.stringify({
        code: '...',
        language: 'java',
        input: ''
    })
})
.then(response => response.text())
.then(data => console.log(data));
```

## 七、关键技术点总结

### 7.1 Spring Boot 3.x变化

| 变化 | 说明 |
|--------|--------|
| javax.* → jakarta.* | Spring Boot 3.x使用Jakarta EE |
| @Resource → @Autowired | 推荐使用@Autowired |
| Duration替代TimeUnit | Apache HttpClient 5.x使用Duration |

### 7.2 Docker Java API使用

| 技术 | 说明 |
|------|--------|
| ResultCallback.Adapter | 解决泛型回调问题 |
| 字符串匹配StreamType | 处理Frame流类型 |
| 容器内文件操作 | 避免跨平台路径问题 |

### 7.3 配置管理

| Spring Boot版本 | 3.2.0 |

## 八、经验教训

### 8.1 依赖管理

1. **版本兼容性**：选择与Spring Boot版本兼容的第三方库
2. **最小化依赖**：只添加必要的依赖，避免冲突
3. **逐步添加**：遇到问题时，逐步添加依赖定位问题

### 8.2 代码调试

1. **错误信息**：仔细阅读错误堆栈，定位根本原因
2. **官方文档**：查阅第三方库的官方文档和示例
3. **社区支持**：搜索GitHub Issues和Stack Overflow

### 8.3 跨平台开发

1. **路径处理**：Windows开发Linux部署时注意路径兼容性
2. **容器化**：使用Docker容器隔离环境，避免依赖宿主机
3. **网络配置**：确保虚拟机网络互通，端口正确开放

### 8.4 测试策略

1. **单元测试**：先测试单个功能，再集成
2. **接口测试**：使用Postman等工具测试API
